# Shift System — Смены и цеха

> Last reviewed: 2026-06-14

## Обзор

Система управления сменами и цехами определяет когда и где работает каждый
сотрудник. Смены назначаются через календарь (ShiftSchedule), а сотрудники
переводятся между сменами с указанием даты вступления. Цеха (Workshop) привязаны
к сменам — сотрудник видит только данные своего цеха.

## Как это работает

### Два уровня: Смена (Shift) и Календарь (ShiftSchedule)

**Shift** — это логическая группа сотрудников (например, "Смена А", "Смена Б"):

- Статусы: `active` / `inactive`
- Сотрудники привязаны через pivot-таблицу `shift_user` с полем `effective_from`
- Один сотрудник — одна смена в данный момент

**ShiftSchedule** — календарь смен по дням:

- Связывает конкретную дату с конкретной сменой
- Используется для определения кто работает сегодня

### Роли, работающие по сменам

```php
// ShiftService::SHIFT_ROLES
['seamstress', 'cutter', 'otk']
```

Только эти роли привязаны к сменному графику. Админы, менеджеры, кладовщики
могут работать независимо.

### Проверка "может ли работать сегодня"

`ShiftService::canWorkToday($user)`:

1. Если роль НЕ в SHIFT_ROLES → **всегда может** работать
2. Если роль в SHIFT_ROLES → проверяет:
    - Текущая смена пользователя (по effective_from <= сегодня)
    - Есть ли эта смена в ShiftSchedule на сегодня
3. Результат: сотрудник может или не может брать заказы

### Открытие/закрытие рабочей смены

**ScheduleService:**

- `openWorkShift()` — фиксирует время начала работы (устанавливает
  `shift_is_open=true`, `start_work_shift=now()`)
- `closeWorkShift()` — фиксирует время окончания
- `isWorkDay()` — проверяет, рабочий ли сегодня день для сотрудника
- `hasWorkDayStarted()` — проверяет, начался ли рабочий день по расписанию

### Middleware RequireOpenShift

Проверяет доступ к операционным роутам:

1. Проверяет настройку `is_enabled_work_shift`
2. Админы и менеджеры проходят без проверки
3. Остальные роли: `user->shift_is_open` должен быть `true`
4. Если нет → редирект на главную с ошибкой

### Перевод сотрудников между сменами

`ShiftService::transferEmployee($userId, $newShiftId, $effectiveFrom)`:

1. Устанавливает `effective_from` для новой смены
2. Старая смена остаётся в истории
3. `getCurrentUsers()` — кто работает сейчас (effective_from <= сегодня)
4. `getIncomingUsers()` — кто придёт в будущем
5. `getOutgoingUsers()` — кто уходит в другую смену

### Цеха (Workshop) и фильтрация

Цеха привязаны к операциям:

- `workshop_id` в `MarketplaceOrderItem` — к какому цеху относится заказ
- `workshop_id` в `Setting` — настройки по цехам
- Фильтрация: сотрудники видят только заказы/данные своего цеха

**Исключение — новые заказы:**

- Новые заказы (status=0) всегда имеют `workshop_id = NULL` до назначения в
  работу
- Виджет «Новые задания на пошив» на дашборде больше не фильтруется по цеху
- `HomeController` вызывает `MarketplaceOrderItemService::new()` без аргумента
- После взятия заказа в работу (`cutter_id` назначается) заказ фильтруется по
  цеху

**Привязка материалов к цехам:**

- Добавлена pivot-таблица `material_workshop` для связи материалов и цехов (
  многие-ко-многим)
- `Workshop::allowedMaterials()` — материалы, доступные для заказа в цехе
- `Material::workshops()` — обратная связь с цехами
- `WorkshopController::edit/update` — UI управления привязкой материалов через
  чекбоксы
- `resources/views/workshops/edit.blade.php` — секция с выбором материалов для
  цеха

### Изоляция рулонов по сменам

Рулоны материалов (`Roll`) привязаны к сменам через `shift_id`. Каждая смена
работает только со своими рулонами:

**Правила доступа:**

1. **Админ и кладовщик** — полный доступ ко всем рулонам (без фильтра по смене)
2. **Швеи, закройщики, ОТК** — только рулоны своей смены
    - Поиск рулона по штрихкоду (`/kiosk/rolls`) фильтруется по `shift_id`
    - Завершение рулона проверяет принадлежность к смене
    - API-поиск рулона (`getRollByCode`) с фильтром по смене
    - Сохранение брака проверяет смену рулона

**Реализация (StickerPrintingController):**

- `rolls()` — при сканировании рулона проверяется `shift_id`
- `completeRoll()` — завершение рулона только своей смены
- `getRollByCode()` — API-метод для поиска с фильтром по смене
- `saveDefects()` — проверка смены при сохранении брака

**Ограничения:**

- Если сотрудник не привязан к смене → ошибка "Вы не привязаны к смене"
- Попытка работы с рулоном другой смены → ошибка "Этот рулон принадлежит другой
  смене"

## Ключевые файлы

- `app/Models/Shift.php` — модель смены (статусы, пользователи, рулоны)
- `app/Models/ShiftSchedule.php` — календарь смен
- `app/Models/Workshop.php` — модель цеха (allowedMaterials — материалы,
  доступные для заказа)
- `app/Services/ShiftService.php` — управление сменами
- `app/Services/ScheduleService.php` — открытие/закрытие рабочих смен
- `app/Http/Controllers/WorkshopController.php` — управление цехами (привязка
  материалов через чекбоксы)
- `app/Services/MovementMaterialToWorkshopService.php` — проверка доступности
  материала цеху
- `app/Models/Schedule.php` — индивидуальное расписание
- `app/Http/Middleware/RequireOpenShift.php` — middleware проверки смены
- `app/Http/Controllers/StickerPrintingController.php` — работа с рулонами
  (изоляция по сменам: rolls(), completeRoll(), getRollByCode(), saveDefects())

- `app/Http/Controllers/SettingController.php` — управление глобальными
  настройками
  (строго workshop_id = NULL)

## Бизнес-правила

- Сменный график актуален только для швей, закройщиков и ОТК
- Перевод между сменами возможен с будущей датой (effective_from)
- Для доступа к операционным роутам смена должна быть открыта
- Админы и менеджеры не привязаны к сменному графику
- Админы и кладовщики имеют полный доступ ко всем рулонам (игнорируют фильтр по
  смене)
- Рулоны изолированы по сменам — каждая смена работает только со своими рулонами
- Ежедневно в 00:01: проверка незакрытых смен и очистка стека заказов

### Управление настройками

- **Глобальные настройки** (`workshop_id = NULL`): управляются через UI
  `/megatulle/setting` (`SettingController::index`/`save`)
- **Цеховые настройки** (`workshop_id ≠ NULL`): не доступны через UI, читаются
  через `Setting::getValue()`/`getValues()` с fallback "цеховая → глобальная"
- **Multi-workshop архитектура**: модель `Setting` поддерживает оба уровня,
  страницы настроек работают ТОЛЬКО с глобальными настройками
