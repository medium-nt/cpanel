# Order Lifecycle — Жизненный цикл заказов маркетплейса

> Last reviewed: 2026-07-17

## Обзор

Заказы с маркетплейсов (Ozon, Wildberries) попадают в систему через
API-синхронизацию каждые 10 минут. Каждый `MarketplaceOrderItem` проходит через
цепочку статусов от "новый" до "готов к отправке". Маршрут заказа зависит от
роли сотрудника (закройщик → швея → ОТК → упаковщик).

## Как это работает

### Статусная машина заказов

Статусы хранятся в `StatusMovement::STATUSES` и `StatusMovement::BADGE_COLORS`:

| Статус                 | Код | Описание                       | Кто берёт в работу   |
|------------------------|-----|--------------------------------|----------------------|
| Новый                  | 0   | Поступил из маркетплейса       | Закройщик, кладовщик |
| В раскрое              | 7   | Закройщик взял в работу        | Закройщик            |
| Выкроено               | 8   | Раскрой завершён               | Швея                 |
| Отшито                 | 4   | Пошив завершён                 | ОТК                  |
| Проверено (маркировка) | 5   | ОТК проверил качество          | Упаковщик/кладовщик  |
| На поставку            | 6   | Готов к отправке               | Упаковщик            |
| Упаковано              | 3   | Товар упакован                 | —                    |
| Брак                   | 11  | Обнаружен дефект               | —                    |
| Готов к отправке       | 13  | Товар на складе, ждёт отправку | —                    |
| Отменён                | 99  | Заказ отменён маркетплейсом    | —                    |
| Отклонён ОТК           | 14  | Заказ отклонён службой ОТК     | —                    |
| Отклонено ОТК          | 14  | Заказ отклонён службой ОТК     | —                    |

### Поток заказа (основной путь)

```
API синхронизация (каждые 10 мин)
  → Новый заказ (status=0)
  → Закройщик берёт в работу (status=7, cutter_id=current_user)
  → Закройщик сдаёт (status=8, cutting_completed_at=now, hanger_id из профиля)
  → Швея берёт в работу (status=4, seamstress_id=current_user)
  → Швея сдаёт → ОТК проверяет (status=5, otk_id=current_user, hanger_id=null)
  → Упаковщик маркирует (status=6)
  → Кладовщик упаковывает (status=3)
  → Товар на складе (status=13)
  → Отправка через поставку (MarketplaceSupply)
```

### Роли и доступ

**MarketplaceOrderItemController::index()** — фильтрация по ролям:

- **Закройщик (cutter):** видит только статус "cutting" (7), может брать заказы
  в раскрой
- **Швея (seamstress):** видит "in_work" (4), может брать только выкроенные
  заказы. **Не видит** новые и заказы в раскрое
- **ОТК (otk):** видит "cut" (8) — проверяет отшитые заказы
- **Кладовщик (storekeeper):** видит "in_work" (4) — помогает с пошивом
- **Менеджер/Админ:** видит все статусы
- **Администраторы:** могут удалять заказы в FBO-поставках (только статус
  "новый", 0)

**Фильтрация по цеху (workshop_id):**

- Сотрудники видят только заказы **своего цеха** (с branch `feature/workshops`)
- Реализовано в `MarketplaceOrderItemService::getFiltered()` через join с
  `marketplace_orders`

**Фильтрация по датам (date_start, date_end):**

- Поля `date_start` и `date_end` в форме фильтрации применимы только к статусу
  «Готовые» (done) для ролей швея и закройщик
- **UI-правило:** поля скрыты для швей/закройщиков на статусах `in_work`,
  `cutting`, `cut` — показываются только при `request('status') === 'done'`
- **Backend-правило:** параметры принудительно удаляются из request при
  `status !== 'done'` для швей/закройщиков — иначе скрытый фильтр silently
  портит выборку
- **Остальные роли** (admin, storekeeper, manager, otk) — даты видны и работают
  для всех статусов
- Реализовано в `MarketplaceOrderItemController::index()` (строки 66-71) и view
  `marketplace_order_items/index.blade.php` (строки 161-180)

**Исключение — новые заказы:**

- Виджет «Новые задания на пошив» на дашборде больше не фильтруется по цеху
- Новые заказы (status=0) всегда имеют `workshop_id = NULL` до назначения в
  работу
- Исправление: `MarketplaceOrderItemService::new()` вызывается без аргумента
- Другие виджеты (toWork, toCutting, urgent, cut) продолжают фильтрироваться по
  цеху

**Права доступа к операциям с заказами:**

- **Администраторы:** могут выполнять все операции (удаление и отвязка заказов)
  через политику `MarketplaceSupplyPolicy::detachOrders()` и `deleteOrders()`
  — проверка через `isAdmin()`

### Удаление заказов

**Только для администраторов:**

- **Удаление доступно только когда поставка в статусе 0 (формируется)**
- Можно удалять заказы по одному или массово все новые заказы в FBO-поставках
- При удалении заказа автоматически удаляются все его позиции (
  `marketplace_order_items`) и история (`marketplace_order_history`) через
  каскадное удаление FK
- Удаление работает только для FBO-поставок, FBS-поставки не затронуты

**Методы удаления:**

- `MarketplaceOrderService::delete()` — удаление одного заказа (только status=0)
- `MarketplaceOrderService::deleteNewOrdersBySupply()` — массовое удаление всех
  заказов в статусе 0 из поставки
- `MarketplaceOrderController::destroyNewBySupply()` — обработчик массового
  удаления
- UI кнопки в раздел "Заказы" на FBO-страницах поставки (
  `show-ozon-fbo.blade.php` и `show-wb-fbo.blade.php`)
- AJAX-обработчик в `public/js/fbo-order-delete.js` — удаление без перезагрузки
  страницы

### Отвязка заказов в FBO-поставках

**Только для администраторов:**

- Отвязка доступна только когда поставка в статусе 13 ("На сборке",
  сформирована)
- Массовая отвязка (НЕ удаление) заказов от поставки: `supply_id = null`, но
  сами заказы остаются в системе
- **Три варианта отвязки:**
    1. **"Убрать не готовые"** (`status = 4`, "Отшито") — заказы без короба
    2. **"Убрать на поставку"** (`status = 6`, "На поставку") — заказы без
       короба
       ✨ **НОВОЕ**
    3. **"Удалить все новые"** (`status = 0`, "Новый") — полное удаление заказов

**"Убрать не готовые" (старая функциональность):**

- Критерий: заказы без короба (`box_id IS NULL`) и в работе (`status = 4`, "
  Отшито")
- Бизнес-смысл: перед отгрузкой убрать заказы, которые ещё не упакованы в короб
- Заказы в коробах (`box_id != null`) всегда остаются в поставке

**"Убрать на поставку" (новая функциональность):**

- Критерий: заказы без короба (`box_id IS NULL`) и статус "На поставку" (
  `status = 6`)
- Бизнес-смысл: перед сборкой убрать заказы, которые ожидают отгрузки
- Только для FBO-поставок (FBS не затрагиваются)

**"Удалить все новые" (удаление):**

- Критерий: только новые заказы (`status = 0`)
- Полное удаление из системы с каскадным удалением позиций и истории
- Только для FBO-поставок в статусе 0 (формируются)

**Методы отвязки:**

- `MarketplaceOrderService::detachNotReadyOrdersBySupply()` — массовое
  отвязывание
  не готовых заказов (`status = 4`)
- `MarketplaceOrderService::detachOnSupplyOrdersBySupply()` — массовое
  отвязывание
  заказов "на поставку" (`status = 6`) ✨ **НОВОЕ**
- `MarketplaceOrderController::detachNotReadyBySupply()` — обработчик массовой
  отвязки не готовых
- `MarketplaceOrderController::detachOnSupplyBySupply()` — обработчик массовой
  отвязки заказов на поставку ✨ **НОВОЕ**
- UI кнопки "Убрать не готовые", "Убрать на поставку", "Удалить все новые" в
  раздел "Заказы" на FBO-страницах поставки
- Форма с confirm + перезагрузка страницы (без AJAX)

### Взятие заказа в работу

`MarketplaceOrderItemService`:

- При взятии заказа назначается `cutter_id` / `seamstress_id` / `otk_id`
- Проверяется дневной лимит: `User::dailyLimitReached()` (лимит в метрах,
  настраивается в Setting)
- Если швея имеет `is_cutter=true` — она может работать как закройщик (
  `User::canSeamstressCut()`)
- **Фильтрация выдачи заказов (`getFilteredItems`):**
    - **Цеховая настройка `orders_filter`** (значения: `all`/`fbo`/`fbs`,
      дефолт `all`) — фильтр по типу исполнения
      `marketplace_orders.fulfillment_type` (FBO/FBS). Применяется ПЕРВОЙ
    - **Персональная настройка `users.orders_priority`** (`all`/`fbo`/`fbo_200`)
      — фильтр по типу исполнения. Применяется ВТОРОЙ
    - Логика пересечения AND: если цех=fbo + швея=fbs → нет заказов (пустое
      пересечение)
    - После фильтрации применяется цеховая настройка `orders_priority`
      (`ozon`/`wb`/`by_date`) для сортировки по маркетплейсу/дате
  - **Цеховая настройка `orders_cluster_priority`** (значения:
    `<marketplace_id>|<cluster>`, напр. `1|Казань`, `2|Коледино`, дефолт
    пусто) — приоритизация FBO-заказов по кластеру. Применяется ПЕРВЫМ в цепочке
    сортировки (CASE WHEN). Сравнение по ОБЕИМ полям (marketplace_id + cluster)
    — защита от коллизий имён кластеров между OZON/WB. ORтогональна
    `orders_priority` и `orders_filter`. **Кластеры маркетплейсов:** OZON (
    marketplace_id=1)
    — поле `cluster` = город-группировка (Казань, Краснодар), много складов
    мапятся
    на 1 кластер. WB (marketplace_id=2) — поле `cluster` пустое, кластером
    служит
    `name` (склад/город). Соответственно, "кластерное значение" = OZON→cluster,
    WB→name.
- **Проверка материалов в цехе (`hasMaterialsInWorkshop`):**
    - Проверяется доступность материалов при взятии заказа (только для ролей
      seamstress и cutter)
    - **Полная матрица «роль × тип материала»:**
      | Тип материала | Константа Material | Когда проверяется |
      |---|---|---|
      | Ткань (Тюль) | TYPE_FABRIC = 1 | Проверяется у закройщиков и швей с
      кроём;
      ПРОПУСКАЕТСЯ для швей без кроя (`seamstressNotCut` — крой делает
      закройщик) |
      | Аксессуары (тесьма и т.п.) | TYPE_ACCESSORY = 2 | Проверяется ТОЛЬКО у
      швей (`isSeamstress()`); ПРОПУСКАЕТСЯ для закройщиков (тесьму пришивает
      швея) |
      | Упаковка | TYPE_PACKAGING = 3 | ПРОПУСКАЕТСЯ всегда; упаковка
      проверяется/
      списывается отдельным потоком упаковщика (`KioskService`,
      `StickerPrintingController`) |
    - **Принцип:** материал проверяется у той роли, которая с ним физически
      работает (ткань — у кроящего, аксессуары — у шьющей швеи, упаковка — у
      упаковщика)

### Автоматический сброс настроек

**Авто-сброс `orders_cluster_priority` при исчерпании очереди кластера:**

- Цеховая настройка `orders_cluster_priority` (формат
  `<marketplace_id>|<cluster>`)
  автоматически сбрасывается в пустое значение, когда все заказы приоритетного
  FBO-кластера сданы в стикеровку (status=5)
- **Логика сброса** (
  `MarketplaceOrderItemService::resetClusterPriorityIfExhausted()`):
    - Принимает `?int $workshopId` — при `int` проверяет конкретный цех, при
      `null` (складской сценарий) обходит ВСЕ цехи с непустой настройкой
    - Проверяет, остались ли заказы этого кластера в очереди (статусы 0,4,7,8)
      — в этом цехе ИЛИ новые нераспределённые (`workshop_id IS NULL`)
    - Если `0` → обнуляет настройку + логирует в канал `system` с текстом
      "Приоритет FBO-кластера сброшен (цех X): Y|Z — все заказы сданы в
      стикеровку"
    - Сбрасывает **только цеховые** настройки (`workshop_id` не NULL),
      глобальную
      не трогает
    - Молчаливый выход если настройка пуста / без разделителя `|`
    - Реализация вынесена в приватный метод
      `resetClusterPriorityForWorkshop(int $workshopId)`
- **Точки триггера** (после перевода заказа в стикеровку, status=5):
    1. `MarketplaceOrderItemController::labeling()` — швея сдаёт заказ
    2. `WarehouseOfItemController` (стр. 242) — кладовщик передаёт в стикеровку

### Дефекты и возвраты

- При обнаружении брака: `status=11`, заполняется `defect_reason`
- Возврат от ОТК: заказ возвращается швее/закройщику на доработку
- Переупаковка: `repacker_id` назначается при перепаковке

**Операции с упаковкой в ОТК (киоск):**

- **Переупаковка (repack):** `/kiosk/item_card/{id}/repack` →
  `StickerPrintingController::processRepack()`
    - Переупаковка товара с изменением упаковки (например, флаер → флаер-пакет)
    - Списывает старую упаковку и новую упаковку с рулона текущей смены
- **Подмена товара (replace):** `/kiosk/process_replace` →
  `StickerPrintingController::processReplace()`
    - Замена одного товара на другой в заказе
    - Списывает упаковку для нового товара с рулона текущей смены
- **Списание упаковки:**
    - `KioskService::hasPackagingMaterials()` — проверяет наличие рулона
      упаковки в
      текущей смене (`Roll::STATUS_IN_WORKSHOP`, `shift_id = $shift->id`)
    - `KioskService::deductPackagingMaterials()` — создаёт Order с
      `shift_id`/`workshop_id` и MovementMaterial с `roll_id`
    - При отсутствии рулона или закрытой смене → `RuntimeException` (транзакция
      откатывается, ошибка логируется в канал `materials`)
- **Бизнес-правило:** рулоны упаковки привязаны к смене (`shift_id`), НЕ к цеху.
  В цехе на смену — один рулон упаковки. Все три точки списания упаковки
  (стикеровка, переупаковка, подмена) используют единый механизм списания с
  рулона через `KioskService`

### Поставки (Supply)

- После упаковки товары объединяются в `MarketplaceSupply`
- Поставка отправляется на маркетплейс через API
- Видео упаковки хранится 60 дней, затем удаляется
- **Лимит 100 заказов на FBS-поставку WB:** Wildberries API ограничивает
  эндпоинт `/api/marketplace/v3/supplies/{supplyId}/orders` до 100 order_id за
  один PATCH-запрос. При попытке добавить 101-й заказ UI блокирует действие с
  сообщением «Достигнут лимит 100 заказов в поставке. Создайте новую поставку.»
  ✨ **НОВОЕ**
- **Механизм отправки:** все order_id отправляются одним PATCH-запросом
  `{"orders": [...]}` вместо N последовательных. Раньше пробивало rate-limit WB.
  ✨ **НОВОЕ**
- **Safety-net:** при `count(order_id) > 100` запрос к WB API не делается, метод
  возвращает false с причиной превышения лимита. ✨ **НОВОЕ**

### Подбор товаров (статус 13 — «На сборке»)

**Статус 13 («На сборке»)** — ключевой статус для логики подбора товаров:

- Заказы в статусе 13 участвуют в сканере подбора (`PickupScan`)
- **Взаимозаменяемость экземпляров:** для заказа status=13 годится **любой**
  экземпляр
  того же артикула (`marketplace_item_id`) со склада (status 11 или 13). Это
  подтверждается
  swap-логикой в `WarehouseOfItemController::labeling`
- **Фильтрация по storage_barcode** (`WarehouseOfItemController::toPickList()`):
    - Кладовщик сканирует `storage_barcode` экземпляра на странице
      `/warehouse_of_item/to_pick_list`
    - Система находит `marketplace_item_id` (артикул) по barcode
    - Фильтрует все заказы, оставляя только те, которые содержат этот товар
    - Query параметр `?storage_barcode=XXX` для фильтрации
    - Если barcode не найден — показывается alert сообщение
    - Кнопка сброса фильтра (красный крестик)
    - Автофокус на input при загрузке страницы
- **Сканер подбора** (`app/Livewire/PickupScan.php`):
    - Кладовщик сканирует `storage_barcode` экземпляра на полке
    - Система берёт `marketplace_item_id` (артикул) и ищет активные заказы
      status=13
    - Подсчитывает «отсканировано/нужно» — живёт только в Livewire-сессии
    - Отсекает «лишние» экземпляры (6-й «Бамбук» при 5 заказов → ошибка)
    - Без записи в БД — только навигационная таблица в рамках сессии
    - **Навигация:** точка входа — кнопка «Сканер подбора» на странице `/warehouse_of_item/to_pick_list` (после «Печать списка»), кнопка «Назад» на сканере → back to to_pick_list
- **Альтернативный путь подбора:** упрощённый маршрут «у полки» без привязки к
  конкретному заказу.
  Текущий процесс: `/warehouse_of_item/to_pick_list` → `/to_pick/{order}` (скан
  по конкретному заказу)

## Бизнес-правила

- Сотрудник видит только заказы своего цеха (workshop_id)
- Швея не может видеть новые заказы — только выкроенные (status=8+)
- Дневной лимит в метрах ограничивает количество работы (seamstress_daily_limit,
  cutter_daily_limit)
- Каждые 10 минут синхронизируются новые и отменённые заказы
- Уведомления при ключевых событиях (взятие заказа, завершение, сканирование) —
  через NotificationService (см. user-management.md: единный шлюз для TG+MAX,
  sync/queued, delay)
- **Правила удаления:** администраторы могут удалять заказы только в
  FBO-поставках,
  только в статусе "новый" (0), только когда поставка в статусе 0 (формируется)
    - Проверка доступа через `MarketplaceSupplyPolicy::deleteOrders()` — только
      `isAdmin()`
- **Фильтрация выдачи заказов швеям:**
    - Цеховая настройка `orders_filter` — фильтр по типу исполнения
      (FBO/FBS/все), применяется ПЕРВОЙ
    - Персональная настройка `users.orders_priority` — фильтр по типу
      исполнения (FBO/FBO-200/все), применяется ВТОРОЙ
    - Логика пересечения AND: если оба фильтра применены и не пересекаются → нет
      заказов
    - После фильтрации цеховая настройка `orders_priority` сортирует по
      маркетплейсу/дате
  - Цеховая настройка `orders_cluster_priority` — приоритизация FBO-заказов по
    кластеру (формат `<marketplace_id>|<cluster>`). Применяется ПЕРВЫМ в цепочке
    сортировки (CASE WHEN), главнее `orders_priority`. Пусто = выключено.
- **Авто-сброс `orders_cluster_priority`:** при переводе последнего заказа
  приоритетного кластера в стикеровку (status=5) настройка автоматически
  обнуляется. Учитывает заказы в очереди (статусы 0,4,7,8) в этом цехе ИЛИ новые
  нераспределённые (`workshop_id IS NULL`). Логирует в канал `system`.
